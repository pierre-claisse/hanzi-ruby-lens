# Research: Text Input UI

**Feature**: 015-text-input-ui
**Date**: 2026-02-17

## R1: View State Management Approach

**Decision**: Use a simple React state union type in the existing `useTextLoader` hook, extended to manage app view state.

**Rationale**: The app has exactly 3 view states (empty, input, saved) plus the existing reading state (when segments exist). A union type `"empty" | "input" | "saved" | "reading"` is the simplest approach. No routing library, no state machine library — just `useState` with derived initial state from `load_text` result.

**Alternatives considered**:
- React Router: Overkill for a single-page desktop app with no URL concerns.
- XState/state machine library: Adds dependency for 4 states with trivial transitions.
- Separate context provider: Unnecessary when useTextLoader already centralizes text state.

## R2: Reusing Existing Tauri Commands vs. New Commands

**Decision**: Reuse `save_text` with `{ rawInput: "...", segments: [] }`. No new Rust code.

**Rationale**: The existing `save_text` command accepts a full `Text` object. Saving with empty segments is semantically correct: the raw input is saved, and segments will be regenerated by LLM in the future (016). The `load_text` command already returns `Option<Text>` — `None` maps cleanly to the empty state, and `Some(Text { segments: [] })` maps to the saved/awaiting-processing state.

**Alternatives considered**:
- New `save_raw_input` command (Rust): Adds complexity with no functional benefit. YAGNI.
- Separate `raw_input` table: Violates the Text aggregate root model. The current `texts` table already has `raw_input` column.

## R3: Sample Text Removal Strategy

**Decision**: Delete `src/data/sample-text.ts`. Modify `useTextLoader` to return `null` instead of falling back to `sampleText`. Update all tests that reference `sampleText`.

**Rationale**: The sample text was a prototyping scaffold. With real input, it becomes misleading — users would see content they didn't enter and can't edit. The empty state (FR-001) explicitly replaces this role.

**Impact on existing tests**:
- `useTextLoader.test.ts`: 3 tests reference `sampleText` — update to expect `null` when DB returns `null` or errors.
- `text-commands.test.ts`: No changes needed (doesn't reference sampleText).
- Integration tests: May need updates if they depend on initial text rendering.

## R4: Textarea Implementation Pattern

**Decision**: Standard `<textarea>` element with Tailwind styling. No rich text editor.

**Rationale**: The input is raw Chinese text — plain text only. No formatting, no markdown, no WYSIWYG. A native textarea handles paste, multiline, CJK input methods, and large content natively. Content-first design (constitution I) means the textarea should be spacious, full-width, with CJK-appropriate font (Noto Sans TC).

**Alternatives considered**:
- CodeMirror/Monaco: Extreme overkill for plain text input.
- contenteditable div: Adds complexity for no benefit. Native textarea already supports CJK IME.

## R5: View State Derivation Logic

**Decision**: Derive initial view state from `load_text` result:

```
load_text returns null          → "empty" state
load_text returns Text, segments empty  → "saved" state (awaiting processing)
load_text returns Text, segments exist  → "reading" state (existing TextDisplay)
```

User actions transition between states:
```
empty   → (click CTA)      → input
input   → (submit)         → saved
input   → (cancel)         → previous state (empty or saved/reading)
saved   → (click edit)     → input
reading → (click edit)     → input
```

**Rationale**: This maps cleanly to the spec's user stories and avoids a separate "view mode" persisted state. The view state is fully derivable from data + user interaction.
